## 1. What are Generics?

Generics were introduced in **Java 5**. They enable types (classes, interfaces, methods) to operate on objects of various types while providing **compile-time type safety**.

### The Problem Without Generics (Unsafe)
Before generics, you used raw types and had to cast manually. Type errors only appeared at **runtime**.

```java
// OLD WAY — UNSAFE
List list = new ArrayList();
list.add("Hello");
list.add(42); // No compile error — dangerous!
String s = (String) list.get(1); // Runtime ClassCastException!
```

### The Solution With Generics (Type-Safe)
```java
// NEW WAY — TYPE-SAFE
List<String> list = new ArrayList<>();
list.add("Hello");
list.add(42); // COMPILE ERROR — caught immediately!
String s = list.get(0); // No cast needed
```

### Key Benefits
1. **Compile-time type safety** — Errors caught before the program runs.
2. **No explicit casting** — Cleaner, more readable code.
3. **Reusability** — Write one class/method that works safely with many types.

## 2. Generic Classes

You can create classes that accept a **type parameter** (usually denoted by a single uppercase letter like `T`).

```java
// T is a placeholder for whatever type the user chooses
class Box<T> {
    private T value;

    public void set(T value) {
        this.value = value;
    }

    public T get() {
        return value;
    }
}
```

**Usage:**
```java
// Now T becomes Integer
Box<Integer> intBox = new Box<>();
intBox.set(42);
int val = intBox.get(); // Returns Integer, no cast needed

// Now T becomes String
Box<String> strBox = new Box<>();
strBox.set("Hello");
String s = strBox.get(); // Returns String, no cast needed
```

You write the `Box` class **once**, but it works safely for any type.

## 3. Generic Methods

You can make a **single method** generic without making the whole class generic. The syntax puts `<T>` **before the return type**.

```java
public class Utils {
    // <T> declares the type parameter
    public static <T> void printArray(T[] arr) {
        for (T item : arr) {
            System.out.print(item + " ");
        }
    }

    // Bounded generic method
    public static <T extends Comparable<T>> T max(T a, T b) {
        return a.compareTo(b) > 0 ? a : b;
    }
}
```

**Usage:**
```java
Integer[] nums = {1, 2, 3};
Utils.printArray(nums); // T becomes Integer

System.out.println(Utils.max(10, 20));       // 20
System.out.println(Utils.max("Apple", "Zebra")); // Zebra
```

### Bounded Type Parameters (`extends`)
Sometimes you want to restrict what types are allowed. You use `extends` to set an **upper bound**.

```java
// T must be Number, or a subclass of Number (Integer, Double, etc.)
class MathBox<T extends Number> {
    private T value;

    MathBox(T value) {
        this.value = value;
    }

    double doubled() {
        // This works because Number guarantees .doubleValue() exists
        return value.doubleValue() * 2;
    }
}

MathBox<Integer> mb = new MathBox<>(5);
System.out.println(mb.doubled()); // 10.0

// MathBox<String> sb = new MathBox<>("Hi"); // COMPILE ERROR
```

**Common bounds:**
- `<T extends Number>` — T must be numeric (Integer, Double, etc.)
- `<T extends Comparable<T>>` — T must support comparison (for sorting, max/min)

## 4. Wildcards (`?`)

Wildcards are used when you don't care about the specific type, or you want to accept a range of types.

### 1. Unbounded Wildcard: `?`
Accepts a list of **any** type.

```java
void printList(List<?> list) {
    for (Object o : list) {
        System.out.println(o);
    }
}

printList(List.of("Alice", "Bob")); // Works
printList(List.of(1, 2, 3));        // Works
```

**Restriction:** You **cannot add** anything to a `List<?>` (except `null`). The compiler doesn't know the actual type, so it blocks adds for safety.

### 2. Upper Bounded Wildcard: `? extends Type`
Accepts any type that **is or extends** the specified type.

```java
// Accepts List<Number>, List<Integer>, List<Double>, etc.
double sumList(List<? extends Number> list) {
    return list.stream().mapToDouble(Number::doubleValue).sum();
}
```

**Rule:** With `extends`, you can **read** items (they are guaranteed to be at least the bound type), but you **cannot add** items (except `null`).

### 3. Lower Bounded Wildcard: `? super Type`
Accepts any type that **is or is a superclass** of the specified type.

```java
// Accepts List<Integer>, List<Number>, List<Object>
void addNumbers(List<? super Integer> list) {
    list.add(1); // Safe!
    list.add(2);
}
```

**Rule:** With `super`, you can **add** items (the list is guaranteed to hold the type or something higher), but when you **read**, you only get `Object`.

### The PECS Mnemonic
*   **P**roducer **E**xtends — Use `? extends` if you only need to **read/get** items.
*   **C**onsumer **S**uper — Use `? super` if you only need to **add/put** items.

### `?` vs `T` — When to use which?
| Feature | `List<?>` (Wildcard) | `List<T>` (Type Parameter) |
|---------|---------------------|---------------------------|
| Meaning | "Some unknown type" | "A specific type I will capture" |
| Can add? | No (except null) | Yes (type is known) |
| Return type | `Object` | `T` (preserves type) |
| Use when | You just need to read/iterate | You need to link input to output |

```java
// Wildcard: Can't return the specific type
public static Object getFirstWildcard(List<?> list) {
    return list.get(0);
}
String s = (String) getFirstWildcard(names); // Manual cast required!

// Type Parameter: Preserves the type
public static <T> T getFirstGeneric(List<T> list) {
    return list.get(0);
}
String s = getFirstGeneric(names); // No cast needed!
```

## 5. Generic Interfaces

Interfaces can be generic. This is very common in Java (e.g., `Comparable<T>`, `List<T>`, `Map<K,V>`).

```java
interface Pair<K, V> {
    K getKey();
    V getValue();
}
```

### Implementing a Generic Interface

**Option 1: Specify types immediately**
```java
class StudentGrade implements Pair<String, Character> {
    private String name;
    private char grade;

    public String getKey() { return name; }
    public Character getValue() { return grade; }
}
```

**Option 2: Stay generic (pass the type up)**
```java
class MyPair<K, V> implements Pair<K, V> {
    private K key;
    private V value;

    public K getKey() { return key; }
    public V getValue() { return value; }
}
```

## 6. Type Erasure

This is the "magic trick" of Java generics. Even though generics give you type safety at **compile time**, the JVM **does not know about generics at runtime**.

When the compiler finishes checking your code, it **erases** all generic type information. This process is called **Type Erasure**.

### How it works:
The compiler replaces all type parameters with their **bound** (usually `Object`).

**Your Code (Compile Time):**
```java
List<String> list = new ArrayList<>();
list.add("Hello");
String s = list.get(0); 
```

**What the JVM actually sees (Runtime):**
```java
List list = new ArrayList(); // <String> is gone!
list.add("Hello");
String s = (String) list.get(0); // Compiler inserted the cast for you!
```

### Why does Java do this?
**Backward Compatibility.** Generics were added in Java 5. By erasing types at runtime, Java 5+ code can still work with old pre-Java-5 libraries without breaking.

### Consequence:
Because types are erased, these are identical at runtime:
```java
List<String> list1 = new ArrayList<>();
List<Integer> list2 = new ArrayList<>();

System.out.println(list1.getClass() == list2.getClass()); // TRUE!
```

| Compile Time | Runtime (after erasure) |
|--------------|------------------------|
| `List<String>` | `List` |
| `Box<Integer>` | `Box` |
| `<T extends Number>` | `T` becomes `Number` |

### Important Restrictions due to Erasure:
*   **Cannot do `new T()`** — JVM doesn't know what class to instantiate.
*   **Cannot use `instanceof` with generics** — `if (obj instanceof List<String>)` is illegal.
*   **Cannot create generic arrays** — `new T[10]` is illegal.

## 7. Exam Q&A

**Q: What are generics? Why are they used?**
**A:** Generics allow types to be parameters, providing compile-time type safety. They eliminate `ClassCastException` at runtime, remove the need for explicit casting, and make code reusable for different types.

**Q: What is the difference between `List<?>` and `List<Object>`?**
**A:** `List<Object>` can hold mixed types (String and Integer in the same list). `List<?>` represents a list of **one specific but unknown** type — it could be `List<String>` or `List<Integer>`, but not a mix. You cannot add elements to a `List<?>` (except `null`).

**Q: Explain bounded type parameters with an example.**
**A:** Upper bound `<T extends Number>` restricts T to Number and its subclasses. Example: `class MathBox<T extends Number>` ensures T has numeric methods like `doubleValue()`.

**Q: What is type erasure in generics?**
**A:** Java removes generic type info at runtime (replaces T with Object or bound type) for backward compatibility. So `List<String>` and `List<Integer>` are the same class at runtime.

**Q: Can you do `new T()` inside a generic class?**
**A:** No. Due to type erasure, T becomes Object at runtime, so the JVM doesn't know what class to instantiate.

**Q: What does PECS stand for?**
**A:** **P**roducer **E**xtends, **C**onsumer **S**uper. Use `? extends` when you only read from a collection, and `? super` when you only write to it.
