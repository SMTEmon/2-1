## 1. What are Code Smells?
Code smells are **indicators** (not bugs) of deeper problems in the code. They are patterns that suggest the code may need refactoring. They don't break functionality but hinder readability, maintainability, and extensibility.

> [!info] Key Concept
> A code smell is like a bad odor in a house. The smell itself isn't a fire, but it suggests there might be rotting garbage or a gas leak that needs fixing.

## 2. Categories of Code Smells

### A. Bloaters — Code that has grown too large
These smells indicate that a class or method has grown out of control.

| Smell | Description | Example | Refactoring Solution |
|-------|-------------|---------|----------------------|
| **Long Method** | A method that does too much (ideally >10-15 lines). | A `processOrder()` method that validates, calculates tax, saves to DB, and emails the user. | **Extract Method**: Break it into `validate()`, `calculateTax()`, etc. |
| **Large Class** | A class with too many fields, methods, or responsibilities. | A `User` class that handles database logic, UI rendering, and business rules. | **Extract Class**: Move related fields/methods to a new class (e.g., `UserProfile`, `UserAuth`). |
| **Primitive Obsession** | Using primitives instead of small objects for concepts. | Using `String` for phone numbers, emails, or currency. | **Replace Data Value with Object**: Create a `PhoneNumber` class with validation logic. |
| **Long Parameter List** | Method with 4+ parameters. | `createUser(name, email, age, address, city, zip, phone)` | **Introduce Parameter Object**: Group them into a `UserDetails` object. |
| **Data Clumps** | Same group of data always appears together. | `x`, `y`, and `z` coordinates passed around together in multiple methods. | **Extract Class**: Create a `Point3D` class. |

### B. Object-Orientation Abusers
These smells indicate that the code is not making full use of object-oriented principles.

| Smell | Description | Example | Refactoring Solution |
|-------|-------------|---------|----------------------|
| **Switch Statements** | Complex `switch`/`if-else` based on type codes. | `if (type == "Circle") ... else if (type == "Square") ...` | **Replace Switch with Polymorphism**: Use subclasses and method overriding. |
| **Temporary Field** | Instance variables only used sometimes; makes class inconsistent. | A `Employee` class with a `currentProject` field that is null when on vacation. | **Extract Class**: Move the field to a `ProjectAssignment` class. |
| **Refused Bequest** | Subclass inherits methods/data it doesn't need (LSP violation). | A `Penguin` inheriting `fly()` from `Bird`. | **Replace Inheritance with Delegation** or fix the hierarchy. |
| **Alternative Classes with Different Interfaces** | Two classes do the same thing with different method names. | `ClientA` has `getName()` and `ClientB` has `fetchName()`. | **Rename Method** or use an Interface to standardize. |

### C. Change Preventers — Hinder modification
These smells make it difficult to change one part of the system without breaking others.

| Smell | Description | Example | Refactoring Solution |
|-------|-------------|---------|----------------------|
| **Divergent Change** | One class changed in many different ways for different reasons. | A `Report` class modified when DB schema changes AND when UI changes. | **Extract Class**: Separate the concerns (e.g., `ReportData` vs `ReportView`). |
| **Shotgun Surgery** | One change requires many small changes in many classes. | Adding a new tax rule requires editing `Invoice`, `Order`, and `Checkout`. | **Move Method/Field**: Consolidate the logic into one class. |
| **Parallel Inheritance Hierarchies** | Every subclass in one hierarchy needs a corresponding subclass in another. | Creating `BigButton` forces you to create `BigButtonRenderer`. | **Move Method/Field** or use **Composition** to break the dependency. |

### D. Dispensable — Unnecessary code/data
Code that serves no purpose or duplicates existing functionality.

| Smell | Description | Example | Refactoring Solution |
|-------|-------------|---------|----------------------|
| **Comments** | Comments that explain WHAT code does (should be self-explanatory). | `// Loop through items` followed by `for (Item i : items)` | **Rename Method/Variable**: Make the code readable so comments aren't needed. |
| **Duplicate Code** | Same code in multiple places. | Copy-pasting a validation logic in three different controllers. | **Extract Method** or **Pull Up Method** to a parent class. |
| **Lazy Class** | Class that doesn't do enough to justify its existence. | A `Utility` class with only one method used once. | **Inline Class**: Merge it into the class that uses it. |
| **Data Class** | Class with only fields, getters, setters, no behavior. | A class that is just a "bag of data". | **Move Method**: Give the class behavior (e.g., `calculateTotal()` instead of just storing numbers). |
| **Dead Code** | Code that is never executed. | Commented-out blocks or unused methods. | **Delete**: Remove it. Version control (Git) keeps the history. |
| **Speculative Generality** | Code written for future use that's not needed now. | Abstract classes created "just in case" we need them later. | **Inline Class/Method**: Simplify until the need actually arises. |

### E. Couplers — Excessive coupling between classes
Classes that are too tightly connected, making them hard to reuse.

| Smell | Description | Example | Refactoring Solution |
|-------|-------------|---------|----------------------|
| **Feature Envy** | Method accesses data of another class more than its own. | A `Phone` object accessing `User` fields to format a number. | **Move Method**: Move the method to the class it envies. |
| **Inappropriate Intimacy** | Classes access each other's private parts too much. | Class A digging into Class B's internal collections. | **Move Method/Field** or use **Encapsulation**. |
| **Message Chains** | `a.getB().getC().getD()` — Law of Demeter violation. | `employee.getDepartment().getManager().getSalary()` | **Hide Delegate**: Add a method `employee.getManagerSalary()`. |
| **Middle Man** | Class that only delegates to another — no real purpose. | `Manager` class that just calls `Employee.doWork()`. | **Remove Middle Man**: Call the target class directly. |

## 3. Refactoring Techniques
Refactoring is the process of restructuring existing code **without changing its external behavior** to improve readability, reduce complexity, and make it easier to maintain.

### Common Techniques

#### 1. Extract Method
**Use when:** A method is too long or does multiple things.
```java
// BEFORE
void printInvoice() {
    System.out.println("=== Invoice ===");
    // ... 20 lines of calculation ...
    System.out.println("Total: " + total);
}

// AFTER
void printInvoice() {
    printHeader();
    printTotal();
}
void printHeader() { System.out.println("=== Invoice ==="); }
```

#### 2. Inline Method
**Use when:** A method body is as clear as the name, or the method is trivial.
```java
// BEFORE
int getRating() { return (moreThanFiveLateDeliveries()) ? 2 : 1; }
boolean moreThanFiveLateDeliveries() { return numberOfLateDeliveries > 5; }

// AFTER
int getRating() { return (numberOfLateDeliveries > 5) ? 2 : 1; }
```

#### 3. Extract Class
**Use when:** One class is doing the work of two (Large Class smell).
```java
// BEFORE
class Person {
    String name;
    String officeAreaCode;
    String officeNumber;
}

// AFTER
class Person { String name; TelephoneNumber telephone; }
class TelephoneNumber { String areaCode; String number; }
```

#### 4. Move Method / Move Field
**Use when:** A method/field is used more by another class than the one it lives in.
```java
// BEFORE (Account uses 'interestRate' but it lives in AccountType)
class Account { double interestRate; }

// AFTER
class Account { 
    AccountType type; 
    double getInterestRate() { return type.getRate(); } 
}
```

#### 5. Rename Method / Variable
**Use when:** The name doesn't reveal its purpose.
```java
// BEFORE
void inv() { ... }

// AFTER
void inviteFriends() { ... }
```

#### 6. Replace Temp with Query
**Use when:** You have a temporary variable storing the result of an expression.
```java
// BEFORE
double basePrice = quantity * itemPrice;
if (basePrice > 1000) return basePrice * 0.95;

// AFTER
if (basePrice() > 1000) return basePrice() * 0.95;
double basePrice() { return quantity * itemPrice; }
```

#### 7. Introduce Parameter Object
**Use when:** You have a long list of parameters.
```java
// BEFORE
void createAccount(String name, String email, String phone, String address) { ... }

// AFTER
void createAccount(CustomerDetails details) { ... }
```

#### 8. Replace Switch with Polymorphism
**Use when:** You have a complex `switch` based on type codes.
```java
// BEFORE
switch (bird.type) {
    case EUROPEAN: return getSpeed();
    case AFRICAN: return getSpeed() - loadFactor;
}

// AFTER
class EuropeanBird extends Bird { double getSpeed() { ... } }
class AfricanBird extends Bird { double getSpeed() { ... } }
```

#### 9. Replace Magic Number with Constant
**Use when:** You have literal numbers with no explanation.
```java
// BEFORE
double potentialEnergy = m * 9.81 * h;

// AFTER
final double GRAVITY = 9.81;
double potentialEnergy = m * GRAVITY * h;
```

#### 10. Encapsulate Field
**Use when:** A field is public and accessible directly.
```java
// BEFORE
class Person { public String name; }

// AFTER
class Person { 
    private String name;
    public String getName() { return name; }
    public void setName(String arg) { name = arg; }
}
```

## 4. Key Design Principles

### Law of Demeter (LoD)
**"Don't talk to strangers."**
A method should only call methods on:
1.  Itself (`this`)
2.  Its own fields
3.  Objects passed as arguments
4.  Objects it creates

*   **Violation:** `employee.getDepartment().getManager().getSalary()`
*   **Fix:** Add a delegation method: `employee.getManagerSalary()`

### Middle Man
A class that exists solely to forward requests to another class.
*   **Bad:** Adds complexity without value.
*   **Good:** If it's part of a pattern like **Proxy**, **Decorator**, or **Adapter**, it adds value (security, logging, translation).
