**Topic:** SOLID Principles - The "L" (LSP)
**Goal:** Last-minute memory triggers for exam answers.

---

## 1. The Core Definition (Memorize This)
> [!info] The Golden Rule
> **"Subtypes must be substitutable for their base types."** 
> *Exam translation:* If Class B extends Class A, you must be able to pass Class B into *any* method expecting Class A **without breaking the program or changing expected behavior.**

**Key Terms to use in your answer:**
*   **Behavioral Consistency:** Subclasses must honor the "contract" of the parent.
*   **Polymorphism:** LSP is what makes polymorphism *safe*.
*   **Client Code:** The code using the object shouldn't need to know if it's a base class or a subclass.

---

## 2. The Primary Example (From the PDF)
*If the exam asks you to "Explain LSP with an example," write this:*

### ❌ The Violation (The Bicycle Problem)
*   **Setup:** `Vehicle` base class has `startEngine()`, `drive()`, and `stop()`.
*   **The Bug:** `Bicycle` extends `Vehicle`. But bikes don't have engines!
*   **The Violation:** `Bicycle` overrides `startEngine()` to throw an `UnsupportedOperationException`.
*   **Why it fails LSP:** The client code expects *all* Vehicles to start an engine. Passing a `Bicycle` crashes the program.

### ✅ The Fix (Refactoring)
*   Create an abstract base `Vehicle` with only `drive()` and `stop()`.
*   Create a `MotorVehicle` subclass that adds `startEngine()`.
*   `Truck` extends `MotorVehicle`.
*   `Bicycle` extends `Vehicle`.
*   *Result:* No class is forced to implement a method it can't fulfill.

## 3. LSP vs. OCP (The `instanceof` Trap)
*Professors love testing the intersection of SOLID principles. Use this to get top marks.*

> [!warning] Exam Trap
> Code can be **Open/Closed (OCP)** but still violate **LSP**.

**The "Code Smell" (Violation):**
If you see a `for` loop with `if (person instanceof Boss)`, **LSP is violated**. 
*   *Why?* Because the client is forced to check the exact type to know what method to call (`doBossStuff()` vs `doPeonStuff()`). The objects are *not* seamlessly substitutable.

**The Solution (Polymorphism):**
1. Add a generic `doStuff()` method to the `IPerson` interface.
2. `Boss` and `Peon` implement `doStuff()` with their own specific logic inside.
3. The client just calls `person.doStuff()` blindly. **No type checking needed!**

---

## 4. Exam Cheat Codes (Rapid Fire Triggers)

If you see any of these in an exam code snippet, **LSP IS VIOLATED. Call it out!**

*   🚨 **Trigger 1:** A subclass throws a *new* Exception that the parent didn't throw (like the Bicycle).
*   🚨 **Trigger 2:** A subclass overrides a method but leaves it completely blank (ignoring the parent's behavior).
*   🚨 **Trigger 3:** Client code uses `instanceof` or `getClass()` to figure out what to do with an object.
*   🚨 **Trigger 4:** *Bonus point for exam:* The "Square inherits from Rectangle" problem. (Changing a Square's width changes its height, breaking the Rectangle's behavior rules).

---
### ✍️ How to structure your exam answer:
1. **Define it:** State Barbara Liskov's rule.
2. **Explain it:** Say "Client code shouldn't know the difference between parent and child."
3. **Show the problem:** Write the Bicycle or `instanceof` example.
4. **Show the fix:** Explain abstracting behaviors (`MotorVehicle`) or using pure polymorphism (`doStuff()`).
